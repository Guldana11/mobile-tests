# Автотесты BNK — отчёт и инженерский план

_Подготовлено по итогам работы над EPIC 0–1 (главный экран + переводы). Для передачи команде разработки приложения и планирования автоматизации._

---

## 1. Где мы сейчас

**Закрыто:** каркас (DriverFactory, BaseTest, retry, кросс-платформа Android/iOS), вся воронка до входа
(язык → приветствие → филиалы/о банке → телефон → пароль → PIN → главный экран), EPIC 0 (главный экран)
и EPIC 1 частично (T-05 переводы между своими счетами, T-06 перевод по номеру телефона).

- **~60 тест-методов**, 20 тест-классов, 16 page-объектов.
- **T-06 доведён до конца на iOS:** форма → «Подтверждение» → «Подтвердить» → SMS-код (мок `0000`) →
  экран статуса операции. Оба кейса зелёные на iOS (проверено вживую на симуляторе iPhone 17 / iOS 26.5).

### Паритет Android / iOS

| | Прогоняется |
|---|---|
| **В обоих** (17 классов) | Smoke, LanguageSelection, Welcome, Branches, More, PhoneLogin, AgreementLink, TermsDocument, PasswordLogin, ForgotPassword, PasswordRecovery, PinCode, MainScreen, SideMenu, BottomNav, PullToRefresh, PhoneTransfer |
| **Только Android** | `AccountDetailTest` (T-01), `TransferTest` (T-05) — на iOS блокируются багами доступности (см. §2) |
| **Только iOS (черновик)** | `TransferIosTest` (T-05 iOS) — доходит до экрана перевода, happy-path до подтверждения — TODO |

**Главный вывод:** отставание iOS — не из-за тестов, а из-за **слабого accessibility-дерева приложения на iOS**.
Это же — самый дешёвый рычаг ускорения всего проекта (см. §2 и §3).

## 2. Отчёт для разработчиков — что починить в приложении

Ниже — проблемы приложения, найденные при автоматизации. Бóльшая часть тормозит именно iOS и решается
**добавлением `accessibilityIdentifier`** на ключевые элементы. Это убирает хрупкие «тапы по координатам»
и открывает iOS-паритет почти без доп. работы со стороны QA.

| # | Проблема | Где | Влияние на тесты | Что сделать | Приор. |
|---|----------|-----|------------------|-------------|--------|
| 1 | **Нет accessibilityIdentifier** у строк счетов, полей перевода, пунктов «Быстрого меню» — в дереве `XCUIElementTypeOther` без имени | iOS, главный экран и переводы | Счета не тапаются по id → координатные тапы с ретраями (флейки). `AccountDetailTest`, `TransferTest` не переносимы на iOS | Проставить стабильные `accessibilityIdentifier` (напр. `account_row_<idx>`, `transfer_amount_field`, `quickmenu_item_<key>`) | **P0** |
| 2 | **Экран ввода телефона без кнопки «Назад»** — нельзя вернуться на Welcome | iOS, логин | Кейс «назад с экрана телефона» скипается (баг приложения) | Добавить навигацию назад | P1 |
| 3 | **Bottom sheet «Быстрое меню» без элемента закрытия** в a11y-дереве и без hardware Back | iOS | Кейс закрытия меню скипается; закрытие — только координатный тап | Добавить кнопку/`accessibilityIdentifier` закрытия | P1 |
| 4 | **Pull-to-refresh контрол не экспонируется** в дереве | iOS, главный экран | Проверка обновления скипается на iOS | Сделать refresh-control видимым для a11y | P2 |
| 5 | **Поле SMS-кода `visible=false`** в дереве (ввод пришлось делать тапами по экранной клавиатуре) | iOS, подтверждение перевода | Хрупкий обходной ввод кода | Экспонировать OTP-поле как доступный TextField | P2 |
| 6 | **Хинт суммы лежит в `value`, а не в `label/text`** | iOS, форма перевода | Поле суммы ищется эвристикой (по хинту, с фолбэком на индекс) | Класть подпись в `label` | P2 |
| 7 | **Клавиатурный Back закрывает sheet «Перевести»** вместо скрытия клавиатуры | Android, перевод | Пришлось закрывать клавиатуру через IME «Done» | Не закрывать sheet по back-клавише клавиатуры | P2 |
| 8 | **Расхождение подписей по платформам:** «Перевод» vs «Переводы внутри банка», «Получит перевод **в** BNK» vs «**на** BNK» | Оба | Локаторы пишутся через CONTAINS/свитчи — лишняя сложность | По возможности унифицировать тексты | P3 |
| 9 | **PDF документов: кнопка Accept есть только на Android** | iOS, документы | Ассерты на Accept — Android-only | Привести флоу к паритету (если задумано одинаково) | P3 |
| 10 | **iOS-флоу восстановления пароля не охарактеризован**, сигнал = алерт «Не удаётся выполнить запись» (видеоидентификация) | iOS | Recovery скипается/идёт по обходному сигналу | Документировать ожидаемый happy-path восстановления | P3 |
| 11 | **Чекбоксы согласия при открытии депозита: accessibility-рамка кнопки смещена ВВЕРХ от видимого/кликабельного кружка** — тап по «центру» элемента мимо цели | iOS, форма депозита | Обычный `mobile: tap` по элементу не переключал чекбоксы (казалось, что iOS-депозит не автоматизируется); решено абсолютным тапом ниже рамки | Совместить a11y frame с видимой зоной чекбокса или дать нативный `Toggle` с `accessibilityIdentifier` | P2 |
| 12 | **Пикер счёта-источника депозита показывает только «Текущий счет» + баланс, без номера счёта** | Оба | Нельзя выбрать конкретный счёт по номеру → берём максимальный по балансу | Добавить номер счёта (или `accessibilityIdentifier`) в строки пикера | P3 |

> **Если делать только одно — делайте №1 (accessibilityIdentifier).** Один этот пункт убирает 3–4 источника
> флейков, переносит `AccountDetailTest` и `TransferTest` на iOS и снижает стоимость iOS-паритета (T-26) примерно вдвое.

## 3. Инженерский план — как закончить быстрее

### 3.1 Доделать начатое (1–2 дня, без зависимостей от разработки)

1. **Закоммитить продолжение T-06** (SMS-код → статус операции) — код готов, iOS зелёный.
2. **Изолировать «денежные» тесты в группу.** Новый кейс T-06 реально проводит перевод (1 000 ₸).
   Пометить TestNG-группой `money` и исключить из частых CI-прогонов, гонять ночью. _(см. §4)_
3. **Допроверить T-06 на Android** — ветка `enterSmsCode` для Android написана «по аналогии», не прогонялась.
4. **Закрыть T-05 iOS** (`TransferIosTest` TODO): «Выберите счёт» → выбрать счёт → сумма → «Перевести» → «Подтверждение».
   _(после фикса №1 станет в разы стабильнее)_

### 3.2 Ускорители инфраструктуры (окупаются на всём остатке)

| Что | Зачем | Эффект |
|-----|-------|--------|
| **T-23** быстрый логин до главного экрана (snapshot-сессия / deeplink вместо полной воронки) | Сейчас каждый кейс авторизованной зоны логинится с нуля (~1.5 мин) | Сокращает время прогона в разы — **самый большой выигрыш по времени** |
| **№1 из §2** accessibilityIdentifier (со стороны разработки) | Убирает координатные тапы | Снимает блок iOS-паритета (T-26) |
| **T-25** cold-restart эмулятора между классами в CI | Деградация эмулятора на длинных прогонах → флейки | Стабильность ночных прогонов |
| **T-24** Allure: шаги+скриншоты на падениях | Сейчас артефакты есть, но без шагов | Быстрее разбор падений |
| **T-27** CI-пайплайн (кнопка/ночь) | Нет автоматического запуска | Регрессия без ручного труда |

### 3.3 Порядок по EPIC (предлагаемый)

```
СЕЙЧАС  ──▶ §3.1 (доделать T-06/T-05) ──▶ T-23 быстрый логин ──┐
                                                               ├──▶ EPIC 1 хвост: T-07 (по карте),
разработка: №1 accessibilityIdentifier ──▶ T-26 iOS-паритет ──┘     T-08 (валидации), T-09 (успех/ошибка)
                                                               │
                                                               ▼
                                          EPIC 2 Платежи ▶ EPIC 3 Карты/счета ▶ EPIC 4–5 ▶ T-27 CI
```

- **Сначала ускорители (T-23 + accessibilityIdentifier), потом масса кейсов** — иначе каждый новый
  кейс наследует медленный логин и хрупкость iOS.
- **T-08 (валидации)** дешёвый и не двигает деньги — хороший «наполнитель» между тяжёлыми happy-path.
- **T-09** фактически уже частично сделан в рамках T-06 (SMS → статус) — переиспользовать, а не писать заново.

### 3.4 Оценка

Из `BACKLOG.md`: всего ~30–35 рабочих дней. **Реалистичное сокращение** при выполнении ускорителей:

- T-23 (быстрый логин) экономит ~20–30% времени прогонов и отладки на всём остатке.
- accessibilityIdentifier снижает T-26 (iOS-паритет) с ~2–3д до ~1–1.5д и убирает регулярные флейки.
- **Итого реалистично: ~22–26 дней** вместо 30–35 при тех же объёмах.

## 4. Риски и рекомендации

- ⚠️ **Денежный тест.** `transferByPhoneCompletesWithSmsCode` проводит реальный перевод (1 000 ₸,
  PRIMARY → ГАУХАР С., оба — тестовые счета с большими балансами). Допустимо для ночной регрессии,
  **но нельзя гонять в каждом PR** — пометить группой и/или сделать обратный перевод в teardown.
- **Координатные тапы** (iOS account-open, закрытие меню) — главный источник флейков. Уйдут после §2 №1.
- **Деградация эмулятора/симулятора** на длинных прогонах — закрывается T-25 (cold-restart в CI).
- **Идемпотентность данных:** для валидаций и истории (T-08, T-17) нужны предсказуемые тестовые счета —
  стоит зафиксировать их в `*.properties` и согласовать с бэкендом тест-окружения.

### Приложение. Что уже проверено вживую в этой сессии
- iOS T-06 полный happy-path (форма → подтверждение → SMS `0000` → статус операции) — **PASSED**.
- Маркеры экранов сняты с реального дерева: SMS = «Введите код» + цифровая клавиатура; статус = «Номер операции» + «Операция в процессе» + получатель + сумма.
- **Депозиты (EPIC 8):** реальное открытие вкладов «Срочный» (1 000 ₸) и «Сберегательный» (15 000 ₸) —
  **PASSED на Android И iOS** (не dry). USD-вклад корректно скипается без фондированного $-счёта.
  2 негатива (`DepositValidationTest`) — зелёные на обеих платформах.
- iOS-специфика формы депозита: источник не пре-выбран (тап «Выберите счет» → счёт); чекбоксы согласия —
  тап ниже a11y-рамки (см. §2 №11); после «Открыть депозит» — лист «Подтверждение» → «Подтвердить».
- **Открытие счёта (EPIC accounts):** нижнее действие главного «Открыть счет» → «Открыть текущий счет»
  (валюта KZT/USD, согласие — текст без чекбокса) → «Продолжить» → счёт создан. **PASSED на Android И iOS**
  (KZT + USD, реально). Негативов нет (нет полей для блокировки submit). Рядом на главном также есть
  действия «Снять деньги» и «Скрытые счета» (пока не покрыты).
- Побочно: устранён корневой «флейк логина» — `appWaitDuration` 60→180 сек в `DriverFactory` (приложение
  на медленном эмуляторе не успевало стартовать за 60 сек → `SessionNotCreated`).
